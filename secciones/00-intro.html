Introducción y puesta en marcha

Visión global del proyecto, tecnologías y cómo levantar el entorno desde cero.

DaWeb Reventas es un marketplace de productos de segunda mano (estilo Wallapop). Esta sección te explica de dónde sale cada tecnología que aparecerá luego, por qué se usa y cómo arrancar el proyecto en tu ordenador para experimentar mientras lees el tutorial.

Para empezar

Asume que no sabes nada. Cada concepto se explica antes de usarlo. Si una palabra te suena rara, búscala en este tutorial con Ctrl + K.

1. ¿Qué es una aplicación web?

Una aplicación web tiene dos partes claramente separadas:

Frontend

Lo que ves y manipulas. Vive en tu navegador (Chrome, Firefox…). Está hecho de:

  • HTML: la estructura (qué hay en la página).
  • CSS: la apariencia (colores, espacios, tipografía).
  • JavaScript: el comportamiento (clicks, llamadas a servidor, animaciones).

En DaWeb el frontend está construido con React, una librería que añade superpoderes a HTML+CSS+JS.

Backend

Lo que no ves. Vive en un servidor remoto. Guarda los datos en una base de datos (MySQL) y responde a peticiones del frontend.

En DaWeb el backend es un proyecto Java de la asignatura ArSo (no lo tocamos en este tutorial) que expone una API REST en http://localhost:8090.

Cómo se hablan

1 Tu navegador carga la web desde http://localhost:5173 (frontend Vite).
▼
2 La web hace una petición HTTP a /api/productos.
▼
3 Vite la reenvía a http://localhost:8090/productos (el backend).
▼
4 El backend consulta la base de datos MySQL y responde con un JSON.
▼
5 El frontend recibe el JSON y pinta los productos.

3. Mapa del proyecto

Abre el proyecto en VS Code. La carpeta importante es daweb/daweb/ (la subcarpeta interna). Esto es lo que hay:

estructura 
daweb/                         ← raíz del repo
└── daweb/                     ← frontend (aquí trabajamos)
    ├── public/                ← archivos estáticos
    ├── src/
    │   ├── api/               ← llamadas al backend
    │   ├── components/        ← piezas reutilizables (Header, Card…)
    │   ├── context/           ← estado global (auth)
    │   ├── pages/             ← una por cada URL
    │   ├── assets/            ← imágenes
    │   ├── App.jsx            ← define las rutas
    │   ├── main.jsx           ← arranca React
    │   ├── index.css          ← estilos globales
    │   └── theme.css          ← variables del tema
    ├── index.html             ← HTML que recibe el navegador
    ├── vite.config.js         ← configuración del servidor dev
    └── package.json           ← dependencias y scripts

4. Cómo arrancar el proyecto

Requisitos

  • Node.js 18 o superior. Comprueba la versión con node --version.
  • Opcional: el backend ArSo corriendo en http://localhost:8090 (si quieres que las llamadas devuelvan datos reales).

Pasos

1 Abre una terminal en daweb/daweb/.
2 La primera vez: npm install (descarga las librerías a node_modules/).
3 npm run dev (arranca el servidor de desarrollo en el puerto 5173).
4 Abre http://localhost:5173/ en el navegador.

Otros comandos

npm run devArranca servidor de desarrollo con recarga en caliente
npm run buildGenera la versión optimizada para producción en dist/
npm run previewSirve dist/ para comprobar cómo quedaría desplegado

6. Ejercicios

Ejercicio 1 Inspeccionar la estructura ★ Muy fácil

Abre daweb/daweb/ en tu editor y responde mentalmente:

  1. ¿Cuántos ficheros hay en src/pages/?
  2. ¿Cuántos en src/api/?
  3. ¿Cuál es la dependencia que más pesa según el orden de package.json?
Ver respuestas

15 páginas (Home, Login, Registro, Productos, DetalleProducto, NuevoProducto, EditarProducto, Perfil, PerfilProductos, PerfilVentas, PerfilCompras, PerfilUsuario, AdminUsuarios, AdminTransacciones, Error404).

7 ficheros en api/: auth.js, categorias.js, client.js, compraventas.js, productos.js, usuarios.js, util.js.

La dependencia principal es react (con su pareja react-dom).

Ejercicio 2 Arrancar el proyecto y modificar algo en caliente ★ Fácil
  1. Ejecuta npm install y luego npm run dev.
  2. Abre http://localhost:5173/.
  3. Sin cerrar el servidor, abre src/pages/Home.jsx y cambia el texto del <h1> (línea ~20) por algo como "Mi web personal". Guarda.
  4. Observa el navegador: el cambio aparece sin recargar.
¿Por qué ha funcionado sin recargar?

Vite tiene Hot Module Replacement (HMR): detecta el cambio en el fichero, recompila sólo lo afectado y se lo manda al navegador a través de un WebSocket. El estado de React se preserva.

Ejercicio 3 Detectar el proxy con DevTools ★★ Intermedio
  1. Con la web abierta, pulsa F12 para abrir las herramientas de desarrollo.
  2. Ve a la pestaña Network / Red.
  3. Recarga (F5) y navega a /productos.
  4. Identifica las peticiones a URLs /api/.... Haz click en una: en Request URL verás http://localhost:5173/api/productos, pero en realidad el servidor que responde es el de 8090.
Solución y pista

Mira el fichero vite.config.js: la sección server.proxy indica que cualquier petición a /api se reenvía a http://localhost:8090 y se le quita el prefijo /api con rewrite.

Ejercicio 4 Sin backend ★★ Intermedio

Si todavía no tienes ArSo corriendo: ¿qué partes de la web funcionan y cuáles no? Pruébalo.

Lo que debería pasar
  • Páginas estáticas como Home (/) y Error 404: funcionan, no piden nada.
  • /productos: aparece la página pero con error o lista vacía (la petición falla).
  • Login: el formulario aparece pero al intentar entrar devuelve error de red.

El frontend es independiente del backend en cuanto a interfaz; sólo necesita el backend para datos.

Truco para la entrevista

Si te preguntan "¿qué pasa cuando un usuario abre la web?", responde con los pasos del flujo: navegador → Vite sirve index.html → carga main.jsx → React monta App → React Router lee la URL → pinta la página correspondiente → la página llama a la API → el proxy redirige al backend.
