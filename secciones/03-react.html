React: componentes y JSX

Por qué React existe, cómo piensa, y cómo se traduce todo a DOM real.

Sincronizar la UI con los datos a mano (DOM crudo, jQuery) es una fábrica de bugs. React resuelve eso con una idea muy potente: describir cómo debería verse la UI para un estado dado, y dejar que la librería se encargue de actualizar el DOM. Eso son los componentes. Aquí los recorremos de la pieza mínima hasta la composición real del proyecto.

1. El problema que resuelve React

Imagina una tarjeta de producto con título, precio y badge "Vendido". Cada vez que el precio cambia, tienes que: localizar el nodo del DOM, leer el viejo valor, escribir el nuevo, actualizar el aria-label, decidir si el badge entra o sale… Con tres componentes funciona; con cien, se hunde.

React te deja escribir UNA función que, dado un objeto producto, devuelve cómo se ve. Si producto cambia, React vuelve a llamar a la función y se encarga de aplicar SOLO los cambios mínimos al DOM real. Tú piensas en "qué", no en "cómo".

2. La pieza mínima: un componente

Un componente es una función JS que devuelve algo que React sabe pintar.

componente mínimo 
function Saludo() {
  return <h1>Hola, mundo</h1>;
}

export default Saludo;

Reglas:

  • Nombre en mayúscula: así React lo distingue de etiquetas HTML.
  • Devuelve UN elemento raíz (envuelve varios en un Fragment <>...</>).
  • Se exporta normalmente con export default.
  • Se usa como una etiqueta: <Saludo />.
Mayúscula = componente · minúscula = etiqueta HTML. Es la pista que React necesita para distinguirlos. <saludo /> busca un tag HTML inexistente; <Saludo /> llama a tu función.

3. JSX: HTML dentro de JS (pero no es HTML)

JSX es sintaxis que parece HTML pero compila a llamadas a React.createElement(...). Es azúcar sintáctico.

Lo que tú escribes

<ProductoCard
  key={p.id}
  producto={p}
/>

Lo que el compilador genera

React.createElement(
  ProductoCard,
  { key: p.id, producto: p }
)

Lo que parece HTML pero NO lo es

HTMLJSXMotivo
class="..."className="..."class es palabra reservada en JS.
for="..."htmlFor="..."for también es reservada.
onclick="..."onClick={...}camelCase y JS, no string.
style="color: red"style={{ color: 'red' }}Objeto JS con propiedades camelCase.
<input><input />Las void elements se autocierran.

Interpolar JS con llaves {}

const nombre = 'Ana';
const edad = 25;

// Interpolar valores
return <h1>Hola {nombre}, tienes {edad} años</h1>;

// Atributos dinámicos
return <img src={imagen} alt={descripcion} />;

// Expresiones (no instrucciones)
return <p>{edad >= 18 ? 'Mayor' : 'Menor'}</p>;

// Llamar funciones
return <p>{nombre.toUpperCase()}</p>;
Dentro de { } sólo van expresiones (cosas que devuelven valor). No if, no for, no const. Para condiciones: ternario o &&.

Renderizado condicional

src/components/ProductoCard.jsx 
{producto.vendido && (
  <Badge pill bg="dark" className="position-absolute">Vendido</Badge>
)}
src/components/Header.jsx 
{usuario ? (
  <button className="header-btn" onClick={salir}>Salir</button>
) : (
  <Link to="/login" className="header-btn">Iniciar sesión</Link>
)}

Las dos reglas: {cond && jsx} renderiza si cond es truthy. {cond ? a : b} elige una rama. Recuerda el gotcha del 0 del módulo anterior.

Listas con .map()

src/pages/Productos.jsx 
<div className="productos-grid">
  {productos.map((p) => (
    <ProductoCard key={p.id} producto={p} />
  ))}
</div>

4. Props: el contrato de un componente

Las props son los argumentos que recibe un componente, igual que los parámetros de una función. Se pasan como atributos JSX.

src/components/ProductoCard.jsx 
import { Card, Badge } from 'react-bootstrap';
import { Link } from 'react-router-dom';
import { etiquetaEstado } from '../api/productos';

function ProductoCard({ producto }) {            // ⬅ destructuring de props
  return (
    <Card className="h-100 card-hover position-relative">
      {producto.vendido && (
        <Badge pill bg="dark" className="position-absolute">Vendido</Badge>
      )}
      <Card.Body className="d-flex flex-column">
        <Card.Title>{producto.titulo}</Card.Title>
        <Card.Text>
          {etiquetaEstado(producto.estado)}
          {producto.lugarDeRecogida?.recogida && ` · ${producto.lugarDeRecogida.recogida}`}
        </Card.Text>
        <h4>{producto.precio} €</h4>
        <Link to={`/productos/${producto.id}`} className="btn btn-lav">Ver detalle</Link>
      </Card.Body>
    </Card>
  );
}

export default ProductoCard;

Y se invoca:

{productos.map((p) => (
  <ProductoCard key={p.id} producto={p} />
))}

Aquí producto={p} envía el objeto. Dentro, { producto } es destructuring; equivale a function ProductoCard(props) { const producto = props.producto; ... }.

Defaults y children

src/components/RutaProtegida.jsx 
function RutaProtegida({ children, soloAdmin = false }) { // ⬅ default
  const { usuario, esAdmin } = useAuth();
  if (!usuario) return <Navigate to="/login" replace />;
  if (soloAdmin && !esAdmin) return <Navigate to="/" replace />;
  return children;
}

children es la prop especial: lo que pones entre las etiquetas de apertura y cierre.

Uso

<RutaProtegida>
  <NuevoProducto />   {/* esto es children */}
</RutaProtegida>

Recepción

function RutaProtegida({ children }) {
  return children; // 
}
Prop sin valor = true. <RutaProtegida soloAdmin> ≡ <RutaProtegida soloAdmin={true}>. Hereda del HTML donde <input required> ≡ required="".

Inmutabilidad de props como contrato

❌ Incorrecto

function Card({ producto }) {
  producto.precio = 0; // muta el objeto del padre
  return <p>{producto.precio}</p>;
}

✅ Correcto

function Card({ producto }) {
  const precioMostrar = Math.floor(producto.precio);
  return <p>{precioMostrar}</p>;
}
SOLID · DIP (Dependency Inversion)

ProductoCard NO importa api/productos.js. Recibe el objeto producto ya construido por su padre. Depende de la abstracción "un objeto con campos titulo/precio/estado/vendido", no del concreto "la lista que devuelve listarProductos".

Beneficio: la misma tarjeta sirve para mostrar resultados de búsqueda, productos de un perfil, productos del admin… Cambia quién la llama, no la tarjeta.

5. Composición: cómo se anidan los componentes

SOLID · SRP (Single Responsibility)

Hoy Header.jsx mezcla tres responsabilidades: navegación + estado de sesión + buscador. Funciona, pero crece mal. Una refactorización SRP partiría en piezas pequeñas:

// Antes (todo junto)
function Header() { /* nav + auth + búsqueda inline */ }

// Después (cada uno con UNA razón para cambiar)
function Header() {
  return (
    <header>
      <HeaderNav />
      <HeaderBusqueda />
      <HeaderUser />
    </header>
  );
}

Si mañana se rediseña el buscador, sólo tocas HeaderBusqueda; el resto queda intocable.

8. Cómo React decide actualizar el DOM

React mantiene en memoria un Virtual DOM (árbol de objetos JS, barato de comparar). Cuando el estado cambia, React vuelve a ejecutar la función del componente y compara los dos árboles.

Analogía: arquitecto que compara dos planos y sólo da instrucciones para mover los tabiques que cambiaron. No derriba el edificio entero.

Re-render ≠ repintar DOM

Cuando React "re-renderiza", ejecuta la función y genera el Virtual DOM. Si el resultado es idéntico al anterior, el DOM real no se toca. Son fases distintas: render (generar VDOM) y commit (parchear DOM).

¿Cuándo re-renderiza?

CausaEjemplo en DaWeb
1. Su estado cambiasetProductos([...]) en Productos.jsx
2. Sus props cambianProductoCard cuando producto cambia
3. Su padre re-renderizaSi Productos re-renderiza, todas sus ProductoCard también (por defecto)

La prop key: cómo rastrea listas

❌ Index como key

{items.map((item, i) => (
  <Item key={i} data={item} />
))}

Si el 0 desaparece, React "reutiliza" el componente del índice 0 para el que era 1. Estado interno mal vinculado.

✅ id estable

{productos.map((p) => (
  <ProductoCard key={p.id} producto={p} />
))}

React identifica EXACTO qué elemento es cuál aunque el orden cambie. Estado interno bien vinculado al dato.

Key estable y única por elemento. Si la lista NUNCA se reordena ni filtra, el índice vale; en cualquier otro caso, usa el id.

9. Preguntas trampa frecuentes

"¿Por qué el header se actualiza al hacer login?"

Respuesta: Header llama a useAuth(). Al hacer login, AuthProvider ejecuta setUsuario(u); eso re-renderiza el Provider y, por cascada, todos sus hijos (Layout, Header, etc.). Header, al re-ejecutarse, lee el nuevo usuario del contexto.

"¿Qué pasa si usas el índice como key?"

Si la lista se reordena, filtra o tiene inserciones/borrados, React confunde componentes. Inputs muestran valor incorrecto, animaciones saltan. En DaWeb el id del producto es la key correcta.

"¿Diferencia entre <div> y <Fragment> (<></>)?"

div añade nodo al DOM. Fragment es invisible: agrupa hijos en JSX sin tocar el DOM. Útil cuando JSX necesita un único raíz pero no quieres div extra (celdas de tabla, donde un div dentro de <tr> es HTML inválido).
