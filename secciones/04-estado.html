Estado y efectos (hooks)

La vida de un componente: cómo recuerda y cómo reacciona al mundo.

Un componente sin estado es una función pura: mismos props ⇒ mismo JSX. En cuanto necesita recordar algo entre interacciones, llega useState. En cuanto necesita hablar con el mundo (fetch, timers, document.title), llega useEffect. Si quiere recordar sin re-renderizar, useRef. Vamos a recorrer esta progresión natural pegada al código del proyecto.

1. Sin estado, sólo dibujo

Recapitulemos la sección 03: un componente es una función. Sin estado, es pura: si le pasas los mismos props, devuelve siempre el mismo JSX. Esto está bien para un ProductoCard que sólo muestra datos. Pero, ¿qué pasa cuando el usuario teclea en un input? La página tiene que recordar el texto. Ahí ya no basta una función pura.

Idea clave

Si modificas una variable normal con =, React no se entera y no redibuja. Tienes que usar useState para que React vigile el cambio y vuelva a llamar a tu componente.

2. useState: la memoria del componente

forma general 
import { useState } from 'react';

function MiComponente() {
  const [valor, setValor] = useState(valorInicial);

  return <button onClick={() => setValor(valor + 1)}>{valor}</button>;
}
  • valor: el estado actual.
  • setValor: la función que avisa a React del cambio y dispara re-render.
  • useState(valorInicial): el valor sólo del primer render.
El estado es un snapshot congelado del render. Llamar a setValor(...) NO actualiza valor dentro del handler actual: avisa para el próximo render. Si necesitas el valor previo, usa la forma funcional setValor(prev => prev + 1).

Ejemplo del proyecto: el buscador del header

src/components/Header.jsx 
function Header() {
  const [isOpen, setIsOpen] = useState(false);    // menú hamburguesa
  const [busqueda, setBusqueda] = useState('');    // texto del buscador

  return (
    <form onSubmit={buscar}>
      <input
        type="search"
        value={busqueda}                                       // controlado
        onChange={(e) => setBusqueda(e.target.value)}           // actualiza estado
      />
    </form>
  );
}

Formularios "controlados": el patrón siempre igual

1 Estado const [x, setX] = useState('').
2 Atributo value={x}.
3 Atributo onChange={e => setX(e.target.value)}.

Estado complejo: un objeto entero con N campos

src/pages/Registro.jsx 
const [datos, setDatos] = useState({
  nombre: '', apellidos: '', correo: '',
  clave: '', fechaNacimiento: '', telefono: '',
});

const cambiar = (e) =>
  setDatos({ ...datos, [e.target.name]: e.target.value }); // spread + clave dinámica
¡No mutes el objeto del estado!

NUNCA hagas datos.nombre = 'Ana' y luego setDatos(datos): React compara por referencia (Object.is); si la referencia es la misma, descarta el re-render. SIEMPRE objeto nuevo con { ...datos, campo: valor }.

CRUD inmutable sobre arrays en estado

El mismo patrón de la sección 02, ahora aplicado al estado:

// C — añadir
setLista([...lista, nuevo]);

// R — leer (no muta)
const item = lista.find(x => x.id === id);

// U — actualizar uno
setLista(lista.map(x => x.id === id ? { ...x, ...cambios } : x));

// D — borrar
setLista(lista.filter(x => x.id !== id));

Lo verás veinte veces. Memorízalo: cada operación devuelve un valor NUEVO y se lo pasas al setter.

Lazy initialization: una función en vez de un valor

Si calcular el valor inicial es costoso (parsear un JWT, leer localStorage), pásalo como una función. React la llama UNA SOLA VEZ.

src/context/AuthContext.jsx 
const [usuario, setUsuario] = useState(usuarioDesdeToken);
// ⬆ Pasamos la REFERENCIA, no usuarioDesdeToken() llamada
// React la ejecuta sólo en el primer render.
useState(fn) ≠ useState(fn()). Sin paréntesis = lazy init (corre una vez). Con paréntesis = corre en CADA render y React descarta el resultado tras el primero. Para inicializadores baratos da igual; para parsing/I/O importa mucho.

3. useEffect: hablar con el mundo

"Efecto secundario" es todo lo que pasa fuera del flujo puro de "dibujar JSX": fetch, suscripciones, timers, modificar document.title… useEffect es el lugar donde meter eso.

forma general 
useEffect(() => {
  // código que se ejecuta DESPUÉS de renderizar (commit)
  return () => {
    // (opcional) limpieza al desmontar o antes de re-ejecutar
  };
}, [dependencias]);

El array de dependencias decide CUÁNDO se ejecuta

FormaCuándo se ejecuta
Sin segundo argumentoTras cada render. Casi nunca quieres esto.
[]Sólo una vez, al montar.
[x, y]Al montar y cada vez que x o y cambien.
Regla mental: "si lo usas dentro, ponlo en deps". Toda variable del scope que el efecto lea y pueda cambiar debe estar en el array. Si no, tendrás stale closure.

Patrón cargando/error/datos — el más común

src/pages/Productos.jsx — fragmento real 
// 1) Cargar categorías UNA VEZ al montar
useEffect(() => {
  categoriasApi.listarCategoriasRaiz()
    .then((lista) => setCategorias(lista ?? []))
    .catch(() => setCategorias([]));
}, []);

// 2) Cargar productos cuando cambien filtros o página
useEffect(() => {
  setCargando(true);
  setError('');
  productosApi
    .listarProductos({ ...filtros, page, size: TAMANYO_PAGINA })
    .then((res) => {
      setProductos(desempaquetar(res, 'productoList'));
      setTotalPaginas(totalPaginasRes(res));
    })
    .catch((err) => setError(err.message))
    .finally(() => setCargando(false));
}, [filtros, page]);

Anatomía:

setCargando(true); setError('');UI muestra spinner. Limpiamos error previo por si el último intento falló.
.then(res => setProductos(...))Si todo va bien, guardamos los datos en el estado. Re-render.
.catch(err => setError(...))Si falla, mostramos el mensaje. Re-render.
.finally(() => setCargando(false))Pase lo que pase, quitamos el spinner. Siempre.

Limpieza: cancelar timers, suscripciones

src/pages/Login.jsx 
const intervaloRef = useRef(null);

useEffect(() => () => {
  // función de limpieza: se ejecuta al desmontar
  if (intervaloRef.current) clearInterval(intervaloRef.current);
}, []);

El patrón () => () => { ... } puede confundir: es una arrow que devuelve otra arrow. La interior es la limpieza.

Cuidado: el callback NO puede ser async

useEffect espera que devuelvas undefined o una función de limpieza. Una async siempre devuelve una Promise, que React ignora. Soluciones:

useEffect(() => {
  productosApi.obtenerProducto(id)
    .then(setProducto)
    .catch((err) => setError(err.message));
}, [id]);
useEffect(() => {
  const cargar = async () => {
    try {
      const p = await productosApi.obtenerProducto(id);
      setProducto(p);
    } catch (err) { setError(err.message); }
  };
  cargar();
}, [id]);

4. useRef: recordar sin re-renderizar

Cuando necesitas guardar un valor que persiste entre renders pero no quieres que su cambio dispare un re-render. Timers, IDs ya procesados, referencias al DOM.

CaracterísticauseStateuseRef
Cambiar el valorsetValor(nuevo)ref.current = nuevo
Dispara re-renderSíNo
Persiste entre rendersSíSí
Cuándo usarDatos que la UI necesita mostrarTimers, IDs auxiliares, refs DOM

Caso real 1: no contar la visualización dos veces

src/pages/DetalleProducto.jsx 
const visualizacionContadaRef = useRef(null);

useEffect(() => {
  setCargando(true);
  productosApi.obtenerProducto(id)
    .then(setProducto)
    .catch((err) => setError(err.message))
    .finally(() => setCargando(false));

  // StrictMode dispara el efecto 2 veces en dev. El ref evita doble contador.
  if (visualizacionContadaRef.current !== id) {
    visualizacionContadaRef.current = id;
    productosApi.sumarVisualizacion(id).catch(() => {});
  }
}, [id]);

Caso real 2: guardar un timer del OAuth popup

src/pages/Login.jsx 
const intervaloRef = useRef(null);

intervaloRef.current = setInterval(() => { ... }, 400);

if (intervaloRef.current) clearInterval(intervaloRef.current);
intervaloRef.current = null;

5. Las reglas de los hooks (no negociables)

  1. Sólo en componentes (o en otros hooks).
  2. Siempre en el nivel superior. NO dentro de if, for, while ni funciones anidadas.
  3. Mismo orden en cada render.
Esto rompe React
if (algo) {
  const [x, setX] = useState(0);   // ❌ MAL
}

React no asocia los hooks por nombre, sino por orden de llamada. En cada render el primer hook es el #0, el segundo es el #1, etc. Si pones un hook dentro de un if, en un render existe (posición 3) y en otro no — React mezcla los estados.

6. Ciclo de vida en una página típica

Sigamos a DetalleProducto en una visita real:

1 Usuario click en "Ver detalle". React monta <DetalleProducto />.
2 Primer render: producto=null, cargando=true. JSX = spinner.
3 useEffect se dispara: pide el producto y suma la visualización.
4 Llega la respuesta. setProducto(...) + setCargando(false) ⇒ re-render.
5 Segundo render: JSX muestra los datos.
6 Si click "Comprar": función async modifica estado y re-renderiza con mensaje de éxito.

7. Reglas finas que el entrevistador explota

7.1 El modelo snapshot y la forma funcional

Trampa clásica 
// ❌ NO suma 3
function sumarTres() {
  setContador(contador + 1);  // encola: 0+1
  setContador(contador + 1);  // encola: 0+1 (contador sigue siendo 0)
  setContador(contador + 1);  // encola: 0+1
}
// resultado: 1

// ✅ Forma funcional: cada uno ve el valor pendiente actualizado
function sumarTres() {
  setContador(prev => prev + 1);
  setContador(prev => prev + 1);
  setContador(prev => prev + 1);
}
// resultado: 3

En DaWeb verás esto en Productos.jsx al sincronizar con la URL: setFiltros(prev => ({ ...prev, descripcion })).

7.2 State batching (React 18)

Varios setState en el mismo handler se agrupan en UN solo re-render, incluso dentro de Promises y setTimeout (a partir de React 18).

const enviar = async (e) => {
  e.preventDefault();
  setError('');           // no re-renderiza aún
  setEnviando(true);      // no re-renderiza aún
  // React agrupa en UN solo re-render
  const res = await login(correo, clave);
  setEnviando(false);
  if (res.ok) navegar('/');
  else setError(res.error);
};

7.3 Stale closure: el bug más sutil

// ❌ filtros queda "rancio": siempre el valor inicial
useEffect(() => {
  productosApi.listarProductos(filtros).then(setProductos);
}, []);                       // ← deps vacías

// ✅ filtros en las deps → efecto se re-ejecuta cuando cambie
useEffect(() => {
  productosApi.listarProductos(filtros).then(setProductos);
}, [filtros]);
SOLID · LSP (Liskov Substitution)

Cualquier componente que reciba producto debe aceptar cualquier objeto con la misma "forma" (titulo, precio, estado, vendido, lugarDeRecogida). Si DetalleProducto y ProductoCard requieren el mismo contrato, puedes intercambiar fuentes (datos del API real, datos mock, datos de admin) sin tocar las tarjetas.

De ahí la importancia de documentar las props o usar PropTypes/TypeScript. Sin contrato claro, LSP se rompe en silencio.

9. Preguntas trampa frecuentes

"¿Por qué no async el callback de useEffect?"

useEffect espera undefined o función de limpieza. Async devuelve Promise → React la ignora → posibles memory leaks. Soluciones: .then() o función interna async.

"¿Qué pasa si haces datos.x='Ana'; setDatos(datos)?"

React no re-renderiza. Compara por referencia con Object.is; misma ref ⇒ "nada cambió". El valor está mutado en memoria pero la UI no se actualiza. Por eso siempre spread.

"¿Cuál es la diferencia entre useState y useRef?"

Cambiar state dispara render; cambiar ref.current no. Usa state para lo que la UI muestra, ref para auxiliares que no afectan a la UI (timers, ids, nodos DOM).

Observa

Cada tarjeta mantiene SU PROPIO contador, porque useState es local a cada instancia del componente.

Ejercicio 2 Cambiar dependencias de un useEffect ★★ Intermedio

En Productos.jsx, ¿qué pasa si cambias las deps de [filtros, page] a []?

Respuesta

Sólo se cargan productos una vez. Cambiar filtros o página NO recargaría. Stale closure.

Ejercicio 3 document.title dinámico ★★★ Avanzado

En DetalleProducto.jsx, añade un useEffect que cambie el título de la pestaña al cargar el producto:

useEffect(() => {
  if (producto?.titulo) {
    document.title = `${producto.titulo} · DaWeb Reventas`;
  }
  return () => { document.title = 'DaWeb Reventas'; };
}, [producto]);
Para qué la limpieza

Al navegar fuera, el título vuelve a "DaWeb Reventas". Sin la limpieza, se quedaría con el último producto.

Ejercicio 4 Detectar mutación accidental ★★★ Avanzado

En Registro.jsx, ¿qué pasa si reemplazas setDatos({ ...datos, [campo]: valor }) por datos[campo] = valor; setDatos(datos)?

Respuesta

El input no se actualiza. setDatos(datos) recibe la misma referencia; React decide "nada cambió" y omite el render. Valor mutado en memoria, UI sin reflejo. Por eso siempre spread crea referencia nueva.
