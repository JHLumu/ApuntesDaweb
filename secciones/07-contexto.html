Context y autenticación

La sesión del usuario, de principio a fin: dónde vive, cómo se crea, cómo caduca.

Cuando hay 15 páginas y todas necesitan saber "¿hay alguien logueado? ¿es admin?", pasar esa información como props sería un infierno. Para eso existe el Context API de React. En DaWeb el contexto guarda el usuario actual y todas las acciones de login/logout/registro. Esta sección es la historia completa de una sesión: del prop drilling hasta la caducidad del JWT, sin saltos.

1. El problema que resuelve Context

Imagina esta jerarquía:

1 App sabe quién es el usuario (acaba de hacer login).
2 App renderiza Layout.
3 Layout renderiza Header.
4 Header necesita el usuario para mostrar el nombre.

Pasarlo por props nivel a nivel se llama prop drilling. Context permite a cualquier descendiente leer el dato sin que los intermedios lo conozcan.

3. La definición del canal: context/useAuth.js

src/context/useAuth.js 
import { createContext, useContext } from 'react';

export const AuthContext = createContext(null);
export const useAuth = () => useContext(AuthContext);

Tres líneas: crea el contexto con valor inicial null y define un hook custom useAuth como atajo. Cualquier función que empieza por use y llama a otros hooks se llama "custom hook" — es la forma idiomática de reutilizar lógica entre componentes.

4. El proveedor: context/AuthContext.jsx

Aquí vive el estado de sesión, las funciones de auth y todo lo relacionado.

src/context/AuthContext.jsx — extracto 
const usuarioDesdeToken = () => {
  const token = getToken();
  if (!token) return null;
  const claims = decodeJwt(token);
  if (!claims || (claims.exp && claims.exp * 1000 < Date.now())) {
    clearToken();
    return null;
  }
  return {
    id: claims.sub,
    nombre: claims.nombre,
    apellidos: claims.apellidos,
    roles: claims.roles ?? [],
  };
};

SOLID · SRP (Single Responsibility)

Hoy AuthProvider mezcla tres responsabilidades: estado (useState), persistencia (localStorage vía setToken/clearToken) y API (llamadas a authApi/usuariosApi). Funciona, pero crece mal. Un refactor SRP extraería sub-hooks:

// useTokenStorage: SÓLO persistencia
function useTokenStorage() { return { getToken, setToken, clearToken }; }

// useAuthApi: SÓLO red
function useAuthApi() { return { login, registrar, actualizarPerfil }; }

// AuthProvider: orquesta los dos
function AuthProvider({ children }) {
  const storage = useTokenStorage();
  const api = useAuthApi();
  const [usuario, setUsuario] = useState(() => userFrom(storage.getToken()));
  // ...
}
SOLID · DIP (Dependency Inversion)

AuthProvider no llama a fetch directamente. Depende de la abstracción authApi.login(), no del concreto "el endpoint /auth/login". Cambiar el endpoint sólo toca api/auth.js, no el Provider.

5. Cómo se cuelga el Provider en la app

src/App.jsx 
function App() {
  return (
    <AuthProvider>
      <BrowserRouter>
        <Layout />
      </BrowserRouter>
    </AuthProvider>
  );
}

Al envolver TODA la app, cualquier descendiente puede usar useAuth(). Si lo invirtieras (BrowserRouter por fuera), las rutas no podrían leer el contexto.

6. Uso desde un componente

src/components/Header.jsx 
import { useAuth } from '../context/useAuth';

function Header() {
  const { usuario, esAdmin, logout } = useAuth();
  const navegar = useNavigate();

  return (
    <nav>
      {esAdmin && <NavLink to="/admin/usuarios">Usuarios</NavLink>}
      {usuario
        ? <button onClick={salir}>Salir</button>
        : <Link to="/login">Iniciar sesión</Link>}
    </nav>
  );
}

8. JWT: qué es y cómo se decodifica

Un JWT es un string con tres partes separadas por puntos:

eyJhbGciOi...  .  eyJzdWIiOiI...  .  3Rk2...
   header           payload            firma
  • Header: algoritmo de firma (p. ej. HS256).
  • Payload: los datos (id, nombre, roles, exp).
  • Firma: garantiza que el backend lo creó (sólo el backend con su clave secreta puede generar una válida).
src/api/client.js — decodificación 
export const decodeJwt = (token) => {
  try {
    const payload = token.split('.')[1];
    const json = atob(payload.replace(/-/g, '+').replace(/_/g, '/'));
    return JSON.parse(json);
  } catch {
    return null;
  }
};
token.split('.')[1]Coge el payload (el segundo trozo).
.replace(/-/g, '+').replace(/_/g, '/')JWT usa Base64URL; atob espera Base64 normal. Reemplazo necesario.
atob(payload)Decodifica Base64 a string JSON.
JSON.parse(json)String → objeto JS con los claims.

Campos (claims) que usa DaWeb

ClaimSignificado
sub"subject": id del usuario
nombrenombre
apellidosapellidos
rolesarray (["USUARIO"] o ["USUARIO","ADMINISTRADOR"])
expcaducidad en SEGUNDOS desde el epoch
exp en JWT viene en SEGUNDOS Unix. JS usa MILISEGUNDOS. Por eso siempre claims.exp * 1000 < Date.now(). Olvidar el *1000 hace que el token caduque "hace 54 años".

9. Login con GitHub: OAuth2 Authorization Code

OAuth2 es un protocolo de autorización. GitHub actúa como proveedor de identidad. El backend es el que realmente "habla" con GitHub; el frontend sólo abre el popup y espera.

src/pages/Login.jsx — loginConGithub 
const loginConGithub = () => {
  const popup = window.open(
    '/oauth2/authorization/github',
    'github-oauth',
    `width=${ancho},height=${alto},left=${x},top=${y}`
  );

  intervaloRef.current = setInterval(() => {
    if (popup.closed) {
      clearInterval(intervaloRef.current);
      return;
    }
    let ruta;
    try { ruta = popup.location.pathname; } catch { return; }
    if (!ruta || !ruta.startsWith('/login/oauth2/code/')) return;

    clearInterval(intervaloRef.current);
    const texto = popup.document.body.innerText || '';
    popup.close();
    const datos = JSON.parse(texto);
    if (datos.token) {
      aceptarToken(datos.token);
      navegar('/');
    }
  }, 400);
};
El try/catch al leer popup.location

Mientras el popup está en github.com, el navegador BLOQUEA el acceso (Same-Origin Policy). El try/catch atrapa la excepción en silencio; el intervalo vuelve a intentar 400 ms después. Cuando el popup vuelve a tu dominio (tras la redirección del backend), el acceso se desbloquea y leemos la URL.

Por qué popup + setInterval (no iframe + onload)

GitHub envía X-Frame-Options: DENY → no se puede meter en un iframe. Y el navegador NO emite eventos cuando un popup cambia de URL en otro origen — la única forma de detectar el regreso es preguntar periódicamente.

10. Persistencia: localStorage vs alternativas

localStoragesessionStorageCookie HttpOnly
PersistenciaHasta que se borreHasta cerrar la pestañaSegún expires
Accesible desde JSSíSíNo
Se envía al backendNo (hay que añadirla al header)NoAutomáticamente
Vulnerable a XSSSíSíNo
DaWeb usa✓ (arso_token)
localStorage = persiste hasta borrarlo, JS lo lee. sessionStorage = muere al cerrar pestaña. Cookie HttpOnly = invisible para JS, viaja sola en cada petición.

DaWeb usa localStorage por simplicidad. Riesgo conocido: si hay XSS, un script atacante puede ejecutar localStorage.getItem('arso_token') y robar el token. La alternativa segura serían cookies HttpOnly, pero requieren credentials: 'include' en fetch + Access-Control-Allow-Credentials en el backend con origen explícito.

11. Caducidad del JWT: la limitación stateless

El JWT es stateless: el backend no guarda nada; verifica la firma en cada petición. Ventaja: escalabilidad (cualquier instancia puede verificar). Limitación: no se puede revocar un token antes de que caduque.

Escenario problema 
1. Admin se logea. JWT con exp = ahora + 24h.
2. Admin abusa. El gestor quiere revocarle acceso.
3. Cambia su rol en la BD a 'USUARIO'.
4. Pero el JWT sigue diciendo "ADMINISTRADOR" en el payload.
5. El admin sigue siendo admin hasta las 24h.

Solución estándar (no en DaWeb):
- Tokens de corta vida (15 min) + refresh tokens
- O lista negra (Redis) de tokens revocados

Qué pasa cuando el JWT caduca durante la sesión

En DaWeb: nada en tiempo real. El estado React sigue mostrando al usuario logueado. La próxima petición a la API devuelve 401, client.js lanza Error, la página muestra el error. NO hay renovación ni logout automático — limitación conocida.

12. Re-renders del Context (lo que el entrevistador explota)

Cada vez que el value del Provider cambia, todos los consumidores de useContext(AuthContext) se re-renderizan, aunque la parte que usan no haya cambiado. En DaWeb es aceptable (renders baratos, pocos cambios). En apps grandes se usa useMemo en el value o se parte en varios contextos.

13. Preguntas trampa frecuentes

"¿Qué pasa si el usuario borra el token de localStorage mientras la app está abierta?"

La app no se entera en tiempo real. Sigue mostrándolo logueado. La próxima petición da 401. Para detectarlo en vivo se usaría window.addEventListener('storage', ...), que se dispara cuando OTRA pestaña modifica localStorage.

"¿Puede un atacante modificar el payload del JWT?"

Puede decodificarlo (Base64 es codificación, no cifrado) y leerlo, pero modificar un solo bit invalida la firma. El backend recalcula la firma y rechaza con 401. Sólo quien tiene la clave secreta del servidor puede generar firmas válidas.

"¿Por qué AuthProvider envuelve BrowserRouter?"

Para que el contexto esté disponible también en rutas, guards y redirecciones. Si fuera al revés, RutaProtegida no podría usar useAuth y useNavigate a la vez.
