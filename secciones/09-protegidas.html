Rutas protegidas y roles

Cómo se evita que un anónimo entre a /perfil y que un usuario normal entre a /admin.

DaWeb tiene tres niveles de acceso: público (cualquiera), autenticado (requiere login) y administrador (rol ADMINISTRADOR). Todo el control vive en un único componente envoltorio: RutaProtegida.jsx. Catorce líneas que se ven en acción en muchas rutas de App.jsx.

1. La idea: un componente "guardián"

En React, un envoltorio inspecciona el estado y decide si renderiza sus hijos o redirige a otra ruta. Es declarativo y elegante.

2. El componente RutaProtegida

src/components/RutaProtegida.jsx 
import { Navigate } from 'react-router-dom';
import { useAuth } from '../context/useAuth';

function RutaProtegida({ children, soloAdmin = false }) {
  const { usuario, esAdmin } = useAuth();

  if (!usuario) return <Navigate to="/login" replace />;
  if (soloAdmin && !esAdmin) return <Navigate to="/" replace />;

  return children;
}

export default RutaProtegida;

Lectura línea a línea

  1. Recibe dos props: children (la página a proteger) y soloAdmin (booleano, por defecto false).
  2. Lee del contexto el usuario actual y si es admin.
  3. Si NO hay usuario → renderiza <Navigate to="/login" replace />. React Router detecta esto y cambia la URL a /login.
  4. Si la ruta exige admin y el usuario no lo es → redirige a la home.
  5. Si supera ambas comprobaciones, renderiza los hijos (la página real).
Sobre replace

<Navigate replace> sustituye la entrada actual del historial. Sin replace, si el usuario pulsa "atrás" volvería a la ruta protegida y se generaría un bucle. Con replace, "atrás" lleva a la página anterior real.

Esto NO es seguridad

La protección de rutas en el frontend es cómoda, no segura. Cualquier usuario malintencionado puede saltarse la comprobación editando el JavaScript en su navegador. La SEGURIDAD real es del backend: rechaza con 401/403 cualquier petición sin token válido o sin rol adecuado. Lo del frontend es sólo para mejor experiencia (que no veas pantallas a las que no tienes acceso).

▼
2 React Router matchea /perfil y monta RutaProtegida.
▼
3 Dentro, useAuth() dice usuario === null.
▼

▼

7. Teoría profunda: lo que el entrevistador sabe

Autenticación vs Autorización: dos conceptos distintos

El JWT hace ambas cosas en DaWeb y es importante distinguirlas:

AutenticaciónAutorización
Pregunta"¿Eres quien dices ser?""¿Tienes permiso para esto?"
En DaWebLa firma criptográfica del JWT. Solo el backend con su clave secreta puede firmar un token válido.El array roles en el payload del JWT. El backend comprueba que el rol incluye el permiso requerido.
EjemploUn usuario con token válido está autenticado.Solo si roles.includes('ADMINISTRADOR') puede acceder a /admin/usuarios.

Un caso donde ambas son insuficientes: si el usuario A intenta editar el producto del usuario B. A está autenticado (tiene JWT válido) y autorizado (tiene rol USUARIO). Pero no tiene permiso sobre ESE producto. El backend debe comprobar que idVendedor === idUsuarioToken en la petición.

Por qué la protección frontend NO es seguridad — técnicamente

El bundle JavaScript de la app es código público. Cualquiera puede:

  1. Abrir DevTools → Sources → buscar y leer el código de RutaProtegida.
  2. Desde la consola, insertar un token manipulado en localStorage (el frontend lo aceptaría, el backend lo rechazaría al verificar la firma).
  3. Usar una herramienta como Postman para llamar directamente a endpoints de la API ignorando el frontend por completo.
La seguridad real está en el backend

El backend ArSo verifica el JWT en cada petición: comprueba la firma, la expiración, y el rol del claims. Si alguien evita RutaProtegida y llama directamente a /api/admin/usuarios sin un JWT de admin válido, el backend devolverá 403. La protección del frontend es experiencia de usuario, no seguridad.

replace en Navigate: el problema del bucle de historial

El historial del navegador es una pila de entradas. push añade encima; replace sustituye la entrada actual:

Sin replace (bug)

Historial: [/, /perfil, /login]
                              ↑ actual

Pulso "Atrás" →
actual = /perfil

React Router monta RutaProtegida
→ usuario === null
→ Navigate to="/login" (sin replace)
→ Historial: [/, /perfil, /login, /login]

Bucle infinito de redirecciones

Con replace (correcto)

Historial: [/, /perfil]
                 ↑ actual

Voy a /perfil → redirige a /login con replace
Historial: [/, /login]
(↑ /perfil sustituido)

Pulso "Atrás" →
actual = / (home)
¡Sin bucle!

Prop booleana en JSX: el atajo de soloAdmin

// Estas tres formas son completamente equivalentes:
<RutaProtegida soloAdmin />
<RutaProtegida soloAdmin={true} />
<RutaProtegida soloAdmin={1 === 1} />

Un atributo JSX sin valor asignado es siempre true. Viene de HTML donde <input required> equivale a <input required="">. En JSX: prop sin valor = prop con valor true.

Cómo RutaProtegida conoce el estado sin recibir props

function RutaProtegida({ children, soloAdmin = false }) {
  const { usuario, esAdmin } = useAuth();  // ← lee del contexto
  // ...
}

useAuth() llama a useContext(AuthContext). Esto funciona porque AuthProvider envuelve toda la app en App.jsx, por encima de BrowserRouter. Por eso cualquier componente en cualquier nivel del árbol puede leer el contexto sin que los intermediarios le pasen la info como prop.

Comparación: protección frontend vs protección real (backend)

qué hace cada capa cuando un USUARIO normal intenta GET /api/usuarios 
Caso A: Usuario navega normal por la UI
─────────────────────────────────────────
  Header.jsx → {esAdmin && ...} → NO se pinta
  ↓
  Usuario no ve el link, no llega a la página → bloqueo de UX (no de seguridad)


Caso B: Usuario escribe /admin/usuarios en la barra de URL
──────────────────────────────────────────────────────────
  RutaProtegida → useAuth() → esAdmin=false
  ↓
   → vuelve a la home → bloqueo de UX


Caso C: Atacante llama a /api/usuarios directamente con su JWT (no admin)
────────────────────────────────────────────────────────────────────────
  fetch('/api/usuarios', { headers: {Authorization: 'Bearer mi-jwt'}})
  ↓
  Vite proxy → Backend ArSo
  ↓
  Spring Security verifica firma del JWT → OK
  ↓
  Comprueba que el rol del JWT incluye ADMINISTRADOR → ❌ NO
  ↓
  HTTP 403 Forbidden → ÚNICO bloqueo de SEGURIDAD REAL


Caso D: Atacante inyecta token forjado en localStorage
──────────────────────────────────────────────────────
  localStorage.setItem('arso_token', 'eyJ.payload-modificado.firma-falsa')
  ↓
  RutaProtegida ve roles=ADMINISTRADOR en el payload local → DEJA PASAR
  ↓
  Pero al llamar /api/admin/usuarios:
  ↓
  Backend recalcula firma del JWT → no coincide con la del token
  ↓
  HTTP 401 Unauthorized → bloqueo de seguridad real
Conclusión del diagrama

Los casos A y B son comodidad. El bloqueo real ocurre en el backend (caso C y D). Sin esa última línea de defensa, todo el frontend es inútil. Por eso a un experto NUNCA le bastará con que muestres RutaProtegida: querrá saber que el backend valida también.

Pregunta trampa del entrevistador

"¿Qué pasaría si un admin es degradado a usuario en el servidor pero su JWT no ha expirado?" — Sigue teniendo el rol ADMINISTRADOR en el payload del JWT hasta que expire. Puede acceder a rutas de admin hasta ese momento. Esta es la limitación fundamental del JWT stateless. La solución completa son tokens de corta vida (15-30 min) + refresh tokens. Con tokens de 24h, el admin degradado mantiene acceso hasta un día.

Pregunta trampa del entrevistador

"¿Por qué hay dos comprobaciones separadas en RutaProtegida en lugar de una sola?" — Porque redirigen a destinos distintos. Un usuario no logueado debe ir a /login para autenticarse. Un usuario logueado sin rol admin ya está autenticado — mandarlo a /login sería confuso. Se le manda a / (home) porque sí puede usarla y no necesita autenticarse de nuevo.

Pregunta trampa del entrevistador

"¿Podría un usuario normal llamar a GET /api/admin/usuarios directamente?" — Desde el frontend con RutaProtegida, no llega a la interfaz. Pero si hace la petición HTTP directamente (Postman, curl) con su JWT de usuario normal, el backend debería devolver 403 porque el JWT no incluye el rol ADMINISTRADOR. Si el backend no comprueba el rol en ese endpoint, sería una vulnerabilidad de autorización en el backend.

9. Ejercicios

Ejercicio 1 Comprobar el flujo de redirección manualmente ★ Muy fácil
  1. Asegúrate de no tener sesión iniciada (en consola: localStorage.removeItem('arso_token'); recarga).
  2. Escribe /perfil en la barra del navegador.
  3. Comprueba que automáticamente apareces en /login.
Ejercicio 2 Proteger una ruta nueva ★★ Intermedio

Recuerdas la ruta /acerca del ejercicio de la sección 05. Protégela para que sólo accedan usuarios autenticados:

<Route path="/acerca" element={
  <RutaProtegida>
    <AcercaDe />
  </RutaProtegida>
} />
Ejercicio 3 Añadir un nuevo rol "MODERADOR" ★★★ Avanzado

Imagina que el backend ahora devuelve un rol nuevo MODERADOR. Adapta el frontend:

  1. En AuthContext.jsx, añade: 
    const esModerador = (usuario?.roles ?? []).includes('MODERADOR');
    Y exponlo en el value del Provider.
  2. Modifica RutaProtegida para aceptar una prop soloModerador: 
    function RutaProtegida({ children, soloAdmin = false, soloModerador = false }) {
  const { usuario, esAdmin, esModerador } = useAuth();
  if (!usuario) return <Navigate to="/login" replace />;
  if (soloAdmin && !esAdmin) return <Navigate to="/" replace />;
  if (soloModerador && !esModerador && !esAdmin) return <Navigate to="/" replace />;
  return children;
}
    Nota cómo damos permiso al admin aunque pidamos sólo moderador: jerarquía.
  3. Usa en alguna ruta nueva: <RutaProtegida soloModerador>...</RutaProtegida>.
Ejercicio 4 Recordar la URL de origen tras el login ★★★ Avanzado

Mejora: si un usuario intenta entrar a /vender sin login, redirige a /login?desde=/vender y, tras login OK, devuélvelo a /vender.

  1. En RutaProtegida, usa useLocation para conseguir la ruta actual: 
    import { Navigate, useLocation } from 'react-router-dom';
// ...
const location = useLocation();
if (!usuario) {
  return <Navigate to={`/login?desde=${encodeURIComponent(location.pathname)}`} replace />;
}
  2. En Login.jsx, lee el parámetro y navega allí tras login: 
    import { useSearchParams } from 'react-router-dom';
// ...
const [sp] = useSearchParams();
const desde = sp.get('desde') || '/';
// Al final del login:
if (res.ok) navegar(desde);
Para la entrevista

Si te preguntan "¿cómo se protege una ruta?", responde con esta estructura: (1) Hay un componente envoltorio RutaProtegida que lee el contexto de autenticación. (2) Si no hay usuario, devuelve <Navigate to="/login">. (3) Si requiere admin y no lo es, devuelve <Navigate to="/">. (4) En App.jsx se envuelve la página dentro del componente. (5) Importante: esto es UX, no seguridad real — la seguridad está en el backend.
