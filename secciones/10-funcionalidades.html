Funcionalidades por página

Las 15 páginas de DaWeb agrupadas por flujo de usuario, con CRUD inline.

Esta sección agrupa las páginas por flujo de usuario, no como un catálogo plano. Ver cómo se conectan las páginas (Home → Productos → DetalleProducto → Compra) revela el diseño mucho mejor que listarlas en orden alfabético. Para cada página tienes: qué hace, dónde está, qué API llama, qué letras de CRUD implementa y ejercicios.

Flujos de usuario y páginas que los componen

Mapeo mental: en cualquier app REST, los "list" y "detail" son R, los formularios "nuevo" son C, los "editar" son U, los botones "borrar" son D. Si esto se cumple en una página y no en otra, hay algo raro.

  • Comprar: ver DetalleProducto (acción) + PerfilCompras (historial)

Qué hace

Estado del componente

const navegar = useNavigate();
const [correo, setCorreo] = useState('');

const intervaloRef = useRef(null);

Submit del formulario

Login.jsx — enviar 
const enviar = async (e) => {
  e.preventDefault();                // ⬅ evita que recargue la página
  setError('');
  setEnviando(true);
  const res = await login(correo, clave);   // función del contexto
  setEnviando(false);
  if (res.ok) navegar('/');
  else setError(res.error || 'No se ha podido iniciar sesión');
};
e.preventDefault()

En HTML estándar, al enviar un form el navegador recarga la página. Como nosotros gestionamos el envío con JS, lo cancelamos.

OAuth con GitHub

Ya explicado en detalle en la sección 07. Resumen: popup → setInterval que vigila cuándo el popup vuelve a tu dominio → leer JSON con el token → aceptarToken.

  setError('La contraseña debe tener al menos 6 caracteres');
  setEnviando(false);
  return;
}

Registro — src/pages/Registro.jsx

Qué hace

Formulario con campos: nombre, apellidos, correo, clave, fechaNacimiento, teléfono. Llama a registrar del contexto (que internamente hace POST /usuarios y, si va bien, login automático).

Estado en un solo objeto

Registro.jsx 
const [datos, setDatos] = useState({
  nombre: '', apellidos: '', correo: '',
  clave: '', fechaNacimiento: '', telefono: '',
});

const cambiar = (e) =>
  setDatos({ ...datos, [e.target.name]: e.target.value });

El truco [e.target.name] permite usar UNA función para TODOS los inputs. Cada input tiene su name y la función modifica esa propiedad concreta del objeto.

Validación nativa de HTML

El formulario usa atributos HTML para validar:

  • required: el campo no puede estar vacío.
  • type="email": el navegador valida formato de correo.
  • type="date": muestra un selector de fecha.
  • minLength={4} en la clave.

Teoría profunda: lo que el entrevistador sabe

e.preventDefault(): por qué es imprescindible en SPAs

El comportamiento por defecto del navegador al enviar un formulario: serializa los campos y hace una petición GET o POST a la URL del atributo action del form (o a la URL actual si no hay action), recargando la página.

En una SPA, esa recarga destruye todo el estado de React en memoria: el usuario logueado, los filtros activos, el historial. e.preventDefault() cancela esa navegación nativa y deja al JS gestionar el envío:

const enviar = async (e) => {
  e.preventDefault();   // cancela la recarga nativa del navegador
  // A partir de aquí: control total del envío con JS
  const res = await login(correo, clave);
  if (res.ok) navegar('/');
  else setError(res.error);
};

El patrón cargando/error/datos: las tres fases de cualquier petición

Una petición HTTP puede tardar desde millisegundos hasta segundos. La UX exige mostrar el estado en cada fase:

// 1. Antes de la petición: mostrar spinner
setCargando(true);
setError('');

productosApi.listarProductos(filtros)
  // 2. Si tiene éxito: mostrar datos
  .then(res => {
    setProductos(desempaquetar(res, 'productoList'));
    setTotalPaginas(totalPaginasRes(res));
  })
  // 3. Si falla: mostrar error
  .catch(err => setError(err.message))
  // 4. Siempre: quitar spinner (¡aunque haya error!)
  .finally(() => setCargando(false));

// En JSX — orden importa:
if (cargando) return <Spinner animation="border" />;
if (error) return <Alert variant="danger">{error}</Alert>;
return <ListaProductos productos={productos} />;
.finally() es crítico

Sin .finally(), si la petición falla, setCargando(false) nunca se llama y el spinner queda girando indefinidamente. .finally() se ejecuta tanto si la petición tuvo éxito como si falló, garantizando que el spinner siempre desaparece.

Optimistic update vs Server refresh: la elección de DaWeb

Después de una acción (comprar, editar), hay dos estrategias para actualizar la UI:

Server refresh (DaWeb usa esto)

// Tras crearCompraventa:
await compraventasApi.crearCompraventa({...});
// Pide el producto de nuevo al servidor
const refrescado = await productosApi.obtenerProducto(id);
setProducto(refrescado); // ahora vendido = true

Pro: La UI siempre muestra lo que está en el servidor. Sin inconsistencias.
Contra: Una petición extra de red. Un pequeño delay visible.

Optimistic update (no implementado)

// Actualiza el estado LOCAL antes de confirmar
setProducto(prev => ({...prev, vendido: true}));
// Luego lanza la petición
crearCompraventa({...})
  .catch(() => {
    // Si falla, revertir
    setProducto(prev => ({...prev, vendido: false}));
  });

Pro: La UI responde instantáneamente.
Contra: Si el servidor falla hay que revertir. Si otro usuario compró al mismo tiempo, puede haber inconsistencia.

Computed property [e.target.name]: un handler para N inputs

El truco central de los formularios de DaWeb:

// Sin el truco: un handler por campo
const cambiarNombre = (e) => setDatos({...datos, nombre: e.target.value});
const cambiarCorreo = (e) => setDatos({...datos, correo: e.target.value});

// Con el truco: UN handler para todos los inputs
const cambiar = (e) =>
  setDatos({ ...datos, [e.target.name]: e.target.value });

// En JSX: el atributo "name" conecta el input con la clave del objeto
<Form.Control name="nombre"   value={datos.nombre}   onChange={cambiar} />
<Form.Control name="correo"   value={datos.correo}   onChange={cambiar} />
<Form.Control name="telefono" value={datos.telefono} onChange={cambiar} />

Los corchetes en una clave de objeto literal ([e.target.name]) se llaman computed property name: la expresión dentro se evalúa en tiempo de ejecución como nombre de la clave. Equivale a:

const obj = { ...datos };
obj[e.target.name] = e.target.value;
// Si e.target.name === 'nombre':  obj.nombre = 'Ana'
// Si e.target.name === 'correo':  obj.correo = 'ana@b.com'

sumarVisualizacion y el useRef que protege de StrictMode

const visualizacionContadaRef = useRef(null);

useEffect(() => {
  productosApi.obtenerProducto(id).then(setProducto);

  // Solo sumar si no hemos contado YA este id en este ciclo de vida
  if (visualizacionContadaRef.current !== id) {
    visualizacionContadaRef.current = id;  // marca como contado
    productosApi.sumarVisualizacion(id).catch(() => {});
  }
}, [id]);

En desarrollo con StrictMode, React monta → desmonta → vuelve a montar cada componente. Eso ejecutaría el useEffect dos veces, sumando dos visualizaciones por visita. El useRef actúa como memoria que NO dispara re-renders:

  • Primera ejecución: ref.current = null ≠ id → suma y guarda el id en el ref.
  • Segunda ejecución (StrictMode): ref.current = id = id → no suma (ya contado).
  • Si el usuario navega a otro producto (id cambia): ref.current = id_viejo ≠ id_nuevo → suma y actualiza el ref.

Pregunta trampa del entrevistador

"¿Por qué delete cambios.clave en el submit del Perfil?" — Si el usuario deja el campo clave vacío y se envía { clave: '' } al backend, el PATCH podría sobreescribir la contraseña con una cadena vacía. delete cambios.clave elimina la propiedad del objeto antes de enviarlo, así el backend solo recibe la clave si el usuario realmente escribió algo nuevo.

Pregunta trampa del entrevistador

"¿Puede un usuario comprar su propio producto?" — El frontend muestra botones de "Editar/Eliminar" en lugar de "Comprar" cuando esMio es true. Pero si hubiera un bug de renderizado o alguien llamara la API directamente, el backend debería rechazar la compra si idComprador === idVendedor del producto. Si el backend no lo valida, sería una vulnerabilidad de lógica de negocio.

Pregunta trampa del entrevistador

"¿Qué diferencia hay entre useRef y una variable local al componente?" — Una variable local se crea de nuevo en cada render y pierde su valor entre renders. useRef persiste entre renders porque React guarda el objeto { current: ... } en memoria mientras el componente esté montado. Además, cambiar .current NO dispara un re-render (a diferencia de useState). Por eso es ideal para "memoria privada" del componente que no afecta a la UI.

Si te piden "implementa una funcionalidad X", el guion es siempre el mismo: (1) ¿Qué página? (2) ¿Qué estado hace falta? (3) ¿Qué función de api/ se llama? (4) ¿Cuándo se llama: en useEffect (al montar/cambiar dep) o en respuesta a un evento (onClick/onSubmit)? (5) ¿Qué muestro mientras carga (Spinner), si hay error (Alert) y si no hay datos? (6) ¿Hay que proteger la ruta? Repasa una página cualquiera y verás que cumple este esqueleto.
