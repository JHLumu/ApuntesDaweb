Vite, proxy y build

La herramienta que orquesta el desarrollo y la build de producción.

Vite es la pieza que pone en marcha el proyecto. En desarrollo te da un servidor con recarga instantánea y un proxy que evita CORS; en producción genera una versión optimizada en dist/. En esta sección entendemos por qué es rápido, qué hace el proxy y qué ocurre al hacer build.

1. ¿Qué hace exactamente un bundler?

Tu navegador NO sabe leer JSX ni resolver import 'react' (que apunta a node_modules/). Necesita un traductor. Eso es un bundler.

Si tu app tiene 100 archivos JS y cada uno importa otros, sin bundler el navegador haría 100 peticiones HTTP. Un bundler los une en uno o pocos archivos, transforma JSX a JS plano, elimina código no usado (tree shaking) y minifica.

BundlerArranque devConfiguración
WebpackLento en proyectos grandesCompleja
ParcelMedioCero config
ViteInstantáneoMínima

Vite es rápido en dev porque NO empaqueta: sirve los ES Modules nativos del navegador, transformando con esbuild (Go, 10-100× más rápido que transformadores JS). En producción sí bundlea con Rollup.

3. vite.config.js en DaWeb

daweb/vite.config.js 
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:8090',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
      },
      '/oauth2': { target: 'http://localhost:8090', changeOrigin: false },
      '/login/oauth2': { target: 'http://localhost:8090', changeOrigin: false },
    },
  },
})
plugins: [react()]Activa JSX, Fast Refresh.
server.proxy['/api']Cuando llega /api/productos, Vite lo reenvía a http://localhost:8090/productos.
changeOrigin: trueModifica el header Host a localhost:8090 para que el backend no se confunda.
rewrite: path => path.replace(/^\/api/, '')Quita el prefijo /api antes de reenviar.
'/oauth2'Sin rewrite: la URL del flujo OAuth con GitHub se conserva.

4. Proxy: el truco que evita CORS

El proxy SOLO existe en dev. En producción no hay Vite. Para que /api/... funcione: o bien usas un proxy de nginx, o bien sirves frontend y backend en el mismo dominio, o bien configuras CORS en el backend.
SOLID · DIP (Dependency Inversion)

El frontend depende de VITE_API_BASE (abstracción), no de http://localhost:8090 (concreto). Cambiar el backend = cambiar una variable; no se toca código de componentes.

// MAL: la URL del backend acoplada al cliente
const URL_BACKEND = 'http://localhost:8090';

// BIEN: variable de entorno + valor por defecto al proxy
const URL_BACKEND = import.meta.env.VITE_API_BASE ?? '/api';

5. Hot Module Replacement (HMR)

Al guardar un cambio en un .jsx, Vite no recarga la pestaña: recompila SOLO el módulo, lo envía por WebSocket y React Fast Refresh lo intercambia preservando el estado.

1 Vite detecta el cambio en el sistema de ficheros.
2 Recompila ese módulo (y dependientes directos) — no toda la app.
3 Envía el módulo nuevo por WebSocket (visible en DevTools → Network → WS).
4 React Fast Refresh lo intercambia sin desmontar el árbol.
5 Estado preservado: contador en 5 sigue en 5 tras cambiar el estilo del botón.
Cuándo HMR NO preserva el estado

Si cambias la estructura del estado (añades/quitas useState, cambias tipo), Fast Refresh tiene que reiniciar el componente para coherencia. Lo verás como recarga parcial del componente.

6. Los tres comandos

npm run dev

Servidor en :5173, HMR, proxy. Sólo dev.

npm run build

Versión final en dist/. Bundlea, minifica, tree shaking, hash en nombres.

npm run preview

Sirve dist/ como un servidor real. Para comprobar la build antes de desplegar.

npm install

Lee package.json y descarga a node_modules/.

7. Pipeline de build paso a paso

Estructura típica de dist/

dist/
├── index.html             ← entry point
├── favicon.svg
└── assets/
    ├── index-AbCdEf12.js  ← código JS minificado
    ├── index-XyZ34uV5.css ← CSS minificado
    └── imagen-Q9w8e7r6.jpg ← imagen optimizada
Hashes en los nombres. Si el contenido cambia, el hash cambia → el navegador descarga el archivo nuevo. Si no cambia, lo cachea para siempre. Esto se llama cache busting.

8. Tree shaking: importa sólo lo que usas

✅ Tree shakeable

// SOLO Button y Card entran al bundle
import { Button, Card } from 'react-bootstrap';
import { BsSearch } from 'react-icons/bs';

❌ Sin tree shaking

// Toda react-bootstrap entra (mucho más grande)
import ReactBootstrap from 'react-bootstrap';
import * as Icons from 'react-icons/bs';

DaWeb usa siempre imports nombrados precisamente para beneficiarse de tree shaking.

9. Variables de entorno: la regla VITE_

Vite expone variables al cliente sólo si empiezan por VITE_. Sin ese prefijo, son sólo del proceso Node (servidor) y NO llegan al bundle del navegador.

.env 
VITE_API_URL=/api      # se expone al cliente
DB_PASSWORD=secreto    # NO se expone (sin prefijo VITE_)
const base   = import.meta.env.VITE_API_URL;  // '/api'
const esDev  = import.meta.env.DEV;            // true en npm run dev
const esProd = import.meta.env.PROD;           // true en npm run build
VITE_ prefix obligatorio. Sin prefijo, Vite ignora la variable. Y todo lo que pongas con prefijo VITE_ acaba en el bundle público: nunca pongas secretos (claves API, contraseñas), sólo URLs y configuración pública.

11. Las dos soluciones en producción (cuando NO hay Vite)

Opción 1: nginx como proxy inverso

server {
  listen 80;
  root /var/www/daweb/dist;
  location / {
    try_files $uri $uri/ /index.html;
  }
  location /api/ {
    proxy_pass http://localhost:8090/;
  }
}

Mismo dominio. Sin CORS. Más simple.

Opción 2: CORS abierto en el backend

Backend en api.ejemplo.com responde con:
  Access-Control-Allow-Origin:
    https://daweb.ejemplo.com
  Access-Control-Allow-Headers:
    Authorization, Content-Type

Frontend cambia API_BASE a:
  https://api.ejemplo.com

Útil cuando backend y frontend viven en hosts distintos.

12. Preguntas trampa frecuentes

"¿Qué pasa si subo dist/ tal cual a un servidor sin configurar?"

Las páginas estáticas funcionan si entras desde /. Si recargas en /productos/42, el servidor busca el fichero físico y da 404. Además, /api/... falla porque no hay proxy.

"¿Por qué Vite es tan rápido vs webpack?"

Vite NO bundlea en dev: sirve los ES Modules nativos uno por uno, transformando JSX al vuelo con esbuild (Go, 10-100× más rápido). Webpack bundlea todo antes de servir. Proyectos grandes: Vite <1s, webpack 30-60s.

"Diferencia entre dev y preview"

dev: HMR, source maps, proxy, sin bundling. preview: sirve dist/ como un estático sin HMR ni proxy. Simula producción. Si algo falla sólo en preview, es bug específico de la build.
